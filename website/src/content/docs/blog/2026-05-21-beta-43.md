## PlanetScale, branched per PR

`alchemy/Planetscale` is the new provider — `MySQLDatabase` and
`PostgresDatabase`, branches, credentials, and a Drizzle
integration that plugs straight into Cloudflare Hyperdrive. The
cluster is owned by one stage; PR previews fork off it with
`Resource.ref(id, { stage })`.

```typescript
const database = stage.startsWith("pr-")
  ? yield* Planetscale.PostgresDatabase.ref("app-db", {
      stage: `staging-${stage}`,
    })
  : yield* Planetscale.PostgresDatabase("app-db", {
      region: { slug: "us-east" },
      clusterSize: "PS_10",
    });
```

## Layers stop leaking `WorkerEnvironment`

An **Infrastructure Layer** is supposed to encapsulate which
cloud is underneath. Until this release, every binding API
method (`kv.get`, `bucket.put`, `db.prepare(...).first()`)
returned an `Effect` whose `R` channel carried
`WorkerEnvironment` — and that requirement propagated up
through any Layer that wrapped the binding.
`WorkerEnvironment` is now resolved once inside each
`*BindingLive` Layer and closed over; the returned methods only
require `Alchemy.RuntimeContext`:

```diff lang="typescript"
-kv.get(key, "json"): Effect.Effect<Job | null, KVNamespaceError, WorkerEnvironment>
+kv.get(key, "json"): Effect.Effect<Job | null, KVNamespaceError, RuntimeContext>
```

`RuntimeContext` is the type-system color for "runs only inside
a deployed Function or Worker" — cloud-agnostic, satisfied
automatically by the runtime, never present at plan or init
time. A `JobService` Layer wrapping KV can now expose
`Effect<Job, _, RuntimeContext>` to consumers without
mentioning Cloudflare at all.
